# CHANGELOG Format Reference

How to read and interpret reinhardt-web CHANGELOG files for migration analysis.

---

## Format Standard

Reinhardt CHANGELOGs follow the [Keep a Changelog](https://keepachangelog.com/en/1.0.0/)
format, auto-generated by release-plz from conventional commits.

### Structure

```markdown
# Changelog

All notable changes to this project will be documented in this file.

## [Unreleased]

## [0.1.0-rc.22] - 2026-04-25

### Added
- New feature description (#123)

### Changed
- Breaking change description (#124)

### Deprecated
- Deprecated API description (#125)

### Fixed
- Bug fix description (#126)

### Removed
- Removed API description (#127)

## [0.1.0-rc.21] - 2026-04-23
...
```

---

## Section Semantics

Each section has specific implications for migration:

| Section | Semantic | Migration Action |
|---------|----------|------------------|
| **Added** | New features introduced | Informational — no action required, optional adoption |
| **Changed** | Behavior or API modifications | **Action required** — often breaking; check affected code. Commits with `!` (e.g., `feat!:`) produce entries here |
| **Deprecated** | APIs marked with `#[deprecated]` | **Should migrate** — still compiles but will be removed in a future version |
| **Fixed** | Bug fixes | Usually no action needed — behavior correction may affect code relying on buggy behavior |
| **Removed** | APIs deleted from public interface | **Action required** — breaking; code using these will not compile |
| **Documentation** | Documentation changes | No app code impact |
| **Maintenance** | Build, CI, chore changes | No app code impact |
| **Testing** | Test infrastructure changes | No app code impact |
| **Styling** | Code formatting changes | No app code impact |

### Priority for migration analysis

1. **Removed** — highest priority, compilation will fail
2. **Changed** — high priority, may cause compilation or behavior changes
3. **Deprecated** — medium priority, compiles but should be addressed
4. **Fixed** — low priority, review if relying on specific behavior
5. **Added** — informational only

## PR Number Extraction

CHANGELOG entries contain PR references in the format `(#NNN)`:

```markdown
### Changed
- Change response format to JSON:API specification (#142)
```

The number `142` refers to the PR in the `kent8192/reinhardt-web` repository.

### Getting PR details

```bash
gh pr view 142 -R kent8192/reinhardt-web --json body,title
```

The PR body often contains:
- Detailed migration instructions
- Before/after code examples
- Rationale for the change
- Links to related issues

### Getting linked issue details

If the PR references issues (e.g., "Fixes #98"):
```bash
gh issue view 98 -R kent8192/reinhardt-web --json body,title
```

Issues provide additional context about why the change was made and may contain
discussion about migration strategies.

## Per-Crate CHANGELOGs

Reinhardt is a workspace with multiple crates. Each crate has its own CHANGELOG:

```
reinhardt/CHANGELOG.md                          # Workspace-level (facade crate)
reinhardt/crates/reinhardt-core/CHANGELOG.md
reinhardt/crates/reinhardt-orm/CHANGELOG.md
reinhardt/crates/reinhardt-database/CHANGELOG.md
reinhardt/crates/reinhardt-auth/CHANGELOG.md
reinhardt/crates/reinhardt-http/CHANGELOG.md
reinhardt/crates/reinhardt-query/CHANGELOG.md
reinhardt/crates/reinhardt-test/CHANGELOG.md
reinhardt/crates/reinhardt-dentdelion/CHANGELOG.md
...
```

### Which CHANGELOGs to read

Focus on crates that the user's application depends on. Check `Cargo.toml` features:
```toml
reinhardt = { version = "0.1.0-rc.22", features = ["auth", "database", "dentdelion"] }
```

This means read CHANGELOGs for:
- `reinhardt` (facade — always read)
- `reinhardt-auth` (from `auth` feature)
- `reinhardt-database` (from `database` feature)
- `reinhardt-dentdelion` (from `dentdelion` feature)
- `reinhardt-core` (always included as base dependency)

## Tag Format

release-plz creates per-crate tags in the format:

```
[crate-name]@v[version]
```

Examples:
- `reinhardt@v0.1.0-rc.22`
- `reinhardt-core@v0.2.0`
- `reinhardt-orm@v0.1.5`

### Using tags for version comparison

To see changes between two versions of a specific crate:
```bash
gh api repos/kent8192/reinhardt-web/compare/reinhardt-core@v0.1.0...reinhardt-core@v0.2.0 \
  --jq '.commits[].commit.message'
```

## Version Header Parsing

Entries are grouped by version header `## [version]`. To extract entries between
two versions:

1. Find the line `## [target_version]` — this is the upper bound
2. Find the line `## [current_version]` — this is the lower bound
3. All content between these two lines (exclusive of the current_version header)
   represents changes that need migration review

### Unreleased section

The `## [Unreleased]` section contains changes not yet in any release. When
upgrading to a specific tagged version, skip the Unreleased section. When
upgrading to HEAD (development), include it.

### Version ordering

Versions appear in reverse chronological order (newest first). When reading
for migration, process from oldest to newest (bottom to top between the
bounds) to understand the chronological sequence of changes.
